工具上架說明

產測開發工具啟動器

← 回到工具清單

這份文件是給「想把自己的工具放上 MT Toolbox」的人看的。整套流程設計成不需要改你的工具本身，Launcher 也不會有任何針對特定工具寫死的邏輯——你只要照著下面的步驟，把 GitLab 準備好、把資訊交給維護者就完成了。

整體流程一覽

  1. 把你的工具 repo 設成 Public（或搬到一個新的 public 專案）
  2. 把 build 好的東西打包成一個 zip
  3. 在 GitLab 建立 Release，把 zip 當附件上傳，用 permalink 網址
  4. 驗證這個 permalink 網址不用登入就能下載
  5. 把必要欄位交給 MT Toolbox 維護者，等對方加進 manifest.json
  6. 之後每次發新版，只要照常發 Release 就好——版本號會自動更新，不用再通知任何人

Step 1：Repo 要能匿名下載

Launcher 是在使用者電腦上執行的小程式，它下載你的工具時不會、也不能幫你登入。所以你的工具 repo 必須設成 Public，Launcher 才抓得到檔案。

如果你的工具目前放在需要登入的私有專案（例如公司內部的開發用 repo）底下，不用把整個私有專案公開，做法是：另外開一個獨立的 public 專案專門放 Release 附件，例如 http://10.118.53.32/tools/<your-project>，跟你私有的開發用 repo 分開。開發還是在私有 repo 進行，只有要發布的 Release 才放到這個 public 專案。

Step 2：把 build 好的東西打包成一個 zip

一個 zip 裡面要包含：

  • 主程式 exe（可以不只一個，見下方「多個進入點」小節）
  • 執行時需要的所有依賴檔案：設定檔（.ini）、資料庫/資料檔（.csv 等）、圖片、DLL 等

不需要包含：原始碼、build 工具本身（例如 PyInstaller 產生的中間檔）。

Launcher 解壓後，會在整個解壓出來的資料夾底下遞迴搜尋你告訴它的 exe 檔名，所以 zip 裡面要不要包資料夾、要包幾層都沒關係，只要：

  • 你提供給維護者的 exe 檔名跟 zip 裡實際的檔名完全一致（含副檔名、大小寫盡量一致）
  • 同一個 zip 裡不要有兩個同名的 exe（不同資料夾也不行），不然 Launcher 會抓到不確定是哪一個

Step 3：建立 GitLab Release，並使用「永久連結」

  1. 打一個 tag，例如 v1.0.0（建議照 SemVer 規則：破壞性變更升 MAJOR、加功能升 MINOR、修 bug 升 PATCH）
  2. 在這個 tag 上建立一個 GitLab Release
  3. 把打包好的 zip 當作 Release 的附件上傳
  4. 之後所有人（包含 Launcher）都要用這個永久連結網址去下載，而不是某個特定版本的連結： 
    http://10.118.53.32/tools/<your-project>/-/releases/permalink/latest/downloads/<你的zip檔名.zip>
    這個網址的意思是「這個專案最新一個 Release 的這個附件」，所以你以後每次發新版，只要照常打新 tag、建新 Release、上傳同名的 zip，這個網址完全不用改，永遠自動指向最新版。

Step 4：驗證匿名下載（很重要，不要跳過）

找一台沒登入的瀏覽器（或直接開無痕視窗）貼上你的 permalink 網址，確認：

  • 不會被導去登入頁（/users/sign_in）
  • 可以直接觸發下載，或至少回應 200

如果你有終端機，也可以直接測試（-L 會跟著轉址）：

curl -sL -o /dev/null -w "%{http_code}\n" "你的permalink網址"

看到 200 才算過關。如果看到 302 或被導到登入頁，代表這個專案還不是 Public，或路徑打錯了——這一步沒測過，Launcher 到使用者端一定會下載失敗。

目前可用的圖示

圖示來自 Lucide （開源、ISC 授權）。目前系統裡已經有這些可以直接用：

  • camera — 相機／影像相關
  • database — 資料庫／模擬資料
  • file-diff — 比對／差異
  • radio — RF／無線訊號
  • clipboard-check — 報告／檢查清單／認證
  • box — 一般設備／打包
  • wrench — 維修／工具

想要清單以外的圖示也可以，跟維護者說一聲，只要 Lucide 官網上找得到的 icon 都能加。

如果你的工具有多個進入點（例如一套工具裡有好幾支獨立的 exe）

不用拆成好幾個工具、好幾份 zip。一份 zip、一個版本號、一個下載連結就好，只要額外告訴維護者每個子程式的：

  • 子程式代號（例如 cam）
  • 顯示名稱（例如「CAM 相機能力偵測」）
  • 對應的 exe 檔名（例如 CAM.exe）

畫面上會變成同一張卡片裡有好幾顆啟動按鈕，各自對應一支 exe，共用同一個版本號跟下載連結。

如果你要放的其實只是一個外部網址（不是可執行的工具）

例如某個內部系統的網頁入口，不需要 zip、不需要 exe，只要給維護者那個網址就好。畫面上會顯示成「前往」按鈕，直接開新分頁連過去，完全不會透過 Launcher。

之後每次發新版，要做什麼？

只要照常打新 tag、建新 Release、上傳新 zip（permalink 網址不用變）就好。不需要再通知維護者更新 manifest.json。

原理：Launcher 每次啟動你的工具前，不是看 manifest.json 裡寫死的版本號，而是直接去問你的 Release permalink「你現在實際指到哪一個 tag」，有新版就自動抓新的下來、換掉舊的再執行。manifest.json 裡的版本號只影響網頁卡片上顯示的字樣，就算過時了也不影響使用者實際抓到、跑到的一定是最新版。

（這個自動判斷機制只認得「GitLab Release permalink」這種下載連結格式。如果你的下載連結不是這個格式，Launcher 會退回去讀 manifest.json 裡寫的版本號，這種情況下發新版還是要麻煩通知維護者更新。）

另外 Launcher 也會順便檢查下載連結的 ETag／檔案大小有沒有變——所以就算你不小心「內容換了但忘記打新 tag」，Launcher 還是會抓到差異、自動重新下載，不會一直卡在舊版本。當然還是建議每次異動都老實打新版號，這只是多一層保險。

上架前檢查清單

  • ☐ Repo（或另外開的 public 專案）設成 Public 了嗎？
  • ☐ zip 裡的 exe 檔名跟你要提供的 exe_name 完全一致嗎？
  • ☐ zip 裡有沒有依賴檔案漏包（ini／csv／DLL／圖片）？
  • ☐ 用沒登入的瀏覽器或 curl 測過 permalink 網址，回應是 200 嗎？
  • ☐ exe 可以在乾淨的資料夾（沒有你開發機上其他殘留檔案）直接執行嗎？

有問題怎麼辦？

照著上面步驟卡關、或不確定怎麼設定 GitLab Release，直接找 MT Toolbox 維護者（real_chang）就好。
